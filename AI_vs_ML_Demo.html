Classic ML — XGBoost / CatBoost
✅ Right tool
Trained on 18 months of labeled transactions. Features: transaction_velocity, geo_distance_home, amount_vs_avg, merchant_category_risk.

Scores every transaction in 4ms. PR-AUC = 0.87. SHAP values explain every declined transaction to the customer and regulator.

Adverse action codes generated automatically from top SHAP features — legally defensible.
4ms latency PR-AUC 0.87 SHAP audit trail
AI / LLM
❌ Wrong tool here
Can reason about fraud in general but cannot score 10,000 transactions per second at 4ms each.

No ground truth labels. Outputs are probabilistic — "this looks suspicious because…" is not a fraud score you can threshold.

No audit trail per decision. Cannot satisfy adverse action requirements under ECOA or Reg B.
200ms+ latency Non-deterministic No audit trail
Classic ML — Gradient Boosting
✅ Right tool
Binary classification: will this customer churn in the next 90 days? Features: login frequency, product usage, support tickets, balance trend.

Outputs a calibrated probability. Top decile gets proactive outreach. ROI is directly measurable — retention rate, revenue saved per intervention.

Model retrained monthly as behavior shifts.
Calibrated probability Monthly retraining Measurable ROI
AI / LLM — as augmentation
⚡ Useful as a layer
LLM can't predict churn from structured data alone. But once the ML model flags a high-risk customer, AI can draft a personalized retention message referencing their actual usage patterns.

Or: AI reads support ticket text to extract sentiment signals that feed into the ML churn model as features.

Pattern: ML decides. AI communicates.
Text → features Draft outreach Augments ML
Classic ML
❌ Not designed for this
Could classify dispute type into a fixed taxonomy (unauthorized / duplicate / merchant error) if you have thousands of labeled examples.

But it can't read and understand a freeform customer narrative, extract dates and amounts, compare them to transaction records, and draft a response letter.

Feature engineering unstructured text into a model is expensive and brittle.
Fixed categories only Requires labels Brittle on free text
AI / LLM
✅ Right tool here
Give the LLM the dispute letter, the transaction records, and a resolution policy. It:
1. Extracts key facts (date, amount, merchant, reason)
2. Classifies the dispute type
3. Checks policy eligibility
4. Drafts a plain-language response letter

Zero task-specific training needed. Works on day one.
Zero training Handles novel cases End-to-end workflow
Classic ML — NER / Rule-based
⚡ Possible but costly
Named Entity Recognition can extract structured fields from documents — but only fields it was trained on, in formats it's seen.

New document format? Retrain the extractor. New field type? Add labeled examples. A bank onboarding 50 document types needs 50 extraction models — or one LLM.

Total maintenance burden scales with document variety.
High accuracy on known formats Breaks on novel layouts
AI / LLM
✅ Right tool here
Upload any W-2, paystub, tax return, or utility bill. Prompt: "Extract: name, employer, annual income, address. Return JSON."

Works on unseen layouts immediately. Handles OCR noise, abbreviations, varied formats. Achieves 90%+ accuracy on extraction tasks with zero fine-tuning.

Output feeds directly into downstream ML models (credit, underwriting).
Zero labeled data Any document type Structured JSON output
Classic ML — Regression / Reinforcement
✅ Right tool
Predict optimal price for each customer segment given elasticity, competitive rates, and risk profile. Continuous target variable → regression.

Model trained on historical pricing decisions and outcomes. A/B tested against control group. KPIs: revenue lift, volume, margin.

Full audit trail — every price derives from an explainable model. Critical for fair lending compliance.
Revenue-optimized A/B testable Audit compliant
AI / LLM
❌ Wrong tool
An LLM can explain pricing strategy in words, but it cannot optimize a continuous pricing function across millions of customers.

No gradient descent. No calibrated outputs. Cannot be A/B tested in a statistically rigorous way.

Fair lending risk: if you can't explain why a specific customer got a specific rate, you have a regulatory problem. "The LLM recommended it" is not an answer.
Non-optimizable Can't A/B test Regulatory exposure
Classic ML — Statistical Tests
✅ Right tool for detection
PSI (Population Stability Index) detects feature distribution drift. KS tests flag label shift. Custom alerts when PR-AUC drops below threshold.

Fully automated, runs on every scoring batch. No human in the loop needed for routine drift detection.

When drift is detected: trigger shadow retraining pipeline.
PSI / KS tests Automated alerts Shadow retraining
AI / LLM — for diagnosis
⚡ Useful for root cause
When the alert fires, an LLM can diagnose why: pull the drift report, the model changelog, recent business events, and synthesize a plain-language explanation for the stakeholder.

"Fraud rate spiked 40% on electronics transactions in Pacific Northwest — correlates with a new card-not-present attack vector reported by FS-ISAC on March 3rd."

ML detects. AI explains.
Root cause synthesis Stakeholder narrative Connects to external context
When to use which — the decision guide
📊 What's your data?
Structured tables, labeled rows→ ML
Free text, documents, language→ AI
Both (text + structured)→ Both
⏱️ What's your latency budget?
Under 50ms (real-time decisions)→ ML
200ms–3s acceptable→ AI ok
Batch overnight→ Either
🏛️ Do you need an audit trail?
Yes — per-decision explainability→ ML
Process-level audit ok→ AI ok
Adverse action codes required→ ML only
🎯 How specific is the task?
One objective, known inputs→ ML
Variable tasks, novel inputs→ AI
Complex reasoning over documents→ AI
📈 Do you have labeled training data?
Yes, thousands of examples→ ML
None (zero-shot)→ AI
A few examples (few-shot)→ AI
🔁 Is the output a number or text?
Score, probability, continuous value→ ML
Language, summary, draft→ AI
Structured JSON from unstructured→ AI
The performance curve — why specificity wins
Performance vs. training data available
What the curve means
ZERO DATA — AI wins by default
A general LLM scores around 0.65–0.70 on most tasks immediately. No labels, no training. This is the floor it gives you for free.
~500–2K LABELS — crossover point
Once you have enough labeled examples on your specific pattern, a trained ML model begins to pull ahead. The crossover is task-dependent — faster for fraud, slower for rare events.
LARGE LABELED DATA — ML dominates
At scale, XGBoost/CatBoost trained on your transactions, your customers, your fraud patterns outperforms any general model. It's optimized for your distribution, not humanity's average.
The real win — combining both
🎯 + 🧠
ML decides. AI explains.
XGBoost scores 4ms real-time. SHAP extracts the top 3 features. LLM converts SHAP values into a plain-language explanation for the customer or regulator.

Result: Machine-speed precision + human-readable accountability.
📍 Fraud, credit decisioning, adverse action
📝 → 📊
AI extracts. ML predicts.
LLM reads a document (paystub, dispute letter, support ticket) and outputs structured JSON. That structured data becomes a feature in your ML model.

Result: Unlock unstructured data as model input — without brittle regex or NER pipelines.
📍 Underwriting, onboarding, risk enrichment
📡 + 💬
ML detects. AI diagnoses.
Statistical drift tests (PSI, KS) fire an alert. LLM receives the drift report, model changelog, and recent business events — and writes a stakeholder summary explaining the "why".

Result: Automated detection + human-readable root cause in seconds.
📍 Model monitoring, incident response
⚡
The thesis: AI raises your floor. A trained model raises your ceiling.
General AI gives you 0.65–0.70 on day one, on almost any problem — for free. That's the floor. As soon as you accumulate labeled data on your specific pattern, a trained ML model pulls ahead. The ceiling is what you build. The combination — specific model for precision, AI for language and reasoning around it — is where the real leverage lives.
